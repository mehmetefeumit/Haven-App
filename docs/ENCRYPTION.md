# Encryption in Haven

A factual inventory of the cryptography Haven uses and the open-source libraries
that provide it. All cryptography runs in the Rust core (`haven-core`) and is
exposed to the Flutter app via `flutter_rust_bridge`; the Dart side performs no
cryptography of its own (it only persists secrets through OS secure storage — see
§5). Versions below are taken from `haven-core/Cargo.lock` and
`haven/rust_builder/Cargo.toml`.

---

## 1. End-to-end group messaging — MLS (Marmot)

Haven encrypts every location update and message to a group using **MLS
(Messaging Layer Security, [RFC 9420](https://www.rfc-editor.org/rfc/rfc9420))**,
as specified by the **Marmot protocol**. The MLS ciphersuite is **`0x0001` —
`MLS_128_DHKEMX25519_AES128GCM_SHA256_Ed25519`**:

| Function | Algorithm | Provided by |
|---|---|---|
| Key encapsulation (KEM) | DHKEM(X25519, HKDF-SHA256) | `x25519-dalek`, `curve25519-dalek` |
| AEAD (message encryption) | AES-128-GCM | `aes-gcm`, `aes` |
| Hash / KDF | SHA-256, HKDF-SHA256 | `sha2`, `hkdf`, `hmac` |
| Signatures | Ed25519 | `ed25519-dalek` |
| Hybrid public-key encryption | HPKE ([RFC 9180](https://www.rfc-editor.org/rfc/rfc9180)) | `hpke-rs` |

- **MLS implementation:** OpenMLS, using its `openmls_rust_crypto` provider —
  https://github.com/openmls/openmls
- **Marmot SDK wrapper (MDK):** https://github.com/parres-hq/mdk
- **Marmot protocol spec:** https://github.com/marmot-protocol/marmot

> Note: `openmls_rust_crypto` also bundles ChaCha20-Poly1305 and ML-KEM
> (post-quantum, via `libcrux-ml-kem`) for *other* MLS ciphersuites. Haven's
> selected ciphersuite is the classical `0x0001` above, so those are present in
> the dependency tree but not exercised by Haven.

## 2. Nostr identity & encrypted events — secp256k1 + NIP-44 / NIP-59

- **Identity & signatures:** secp256k1 Schnorr signatures
  ([BIP-340](https://github.com/bitcoin/bips/blob/master/bip-0340.mediawiki)) —
  `secp256k1` (Rust bindings to Bitcoin Core's libsecp256k1), `bitcoin_hashes`.
- **Payload encryption ([NIP-44](https://github.com/nostr-protocol/nips/blob/master/44.md) v2):**
  ChaCha20 (256-bit) for confidentiality + HMAC-SHA256 for authentication, with the
  conversation key derived from a secp256k1 ECDH shared secret via HKDF-SHA256 —
  `chacha20`, `hmac`, `sha2`, `hkdf`.
- **Sealed delivery ([NIP-59](https://github.com/nostr-protocol/nips/blob/master/59.md)
  gift wrap, kind 1059):** events are wrapped with NIP-44 using single-use ephemeral
  keys.

Implementation: `nostr` / `nostr-sdk` (rust-nostr) — https://github.com/rust-nostr/nostr
· secp256k1 bindings — https://github.com/rust-bitcoin/rust-secp256k1 (wrapping
https://github.com/bitcoin-core/secp256k1).

## 3. Local database at rest — SQLCipher

The MLS group state and circle metadata database is encrypted with **SQLCipher**.
SQLCipher's defaults are AES-256-CBC for page data, HMAC-SHA512 for per-page
authentication, and PBKDF2-HMAC-SHA512 for key derivation
([SQLCipher design](https://www.zetetic.net/sqlcipher/design/)). The underlying
cipher primitives are provided by **OpenSSL** (statically vendored into the build).
The database key is generated by MDK and held in the OS keyring (§5).

- SQLCipher — https://github.com/sqlcipher/sqlcipher
- `rusqlite` (SQLite bindings) — https://github.com/rusqlite/rusqlite
- `libsqlite3-sys` with `bundled-sqlcipher-vendored-openssl` — https://github.com/rusqlite/rusqlite/tree/master/libsqlite3-sys
- OpenSSL — https://github.com/openssl/openssl · Rust `openssl-sys` — https://github.com/sfackler/rust-openssl

## 4. Transport — TLS (WebSocket to Nostr relays)

Connections to Nostr relays use WebSocket-over-TLS (`wss://`). TLS is provided by
**rustls** with the **ring** crypto backend; certificate verification uses
`rustls-webpki` with `webpki-roots`, over `tokio-tungstenite`.

- rustls — https://github.com/rustls/rustls
- ring — https://github.com/briansmith/ring
- tokio-tungstenite — https://github.com/snapview/tokio-tungstenite

## 5. Key storage & secret handling (OS-provided)

Haven does not implement its own at-rest key encryption for secrets; it relies on
platform secure storage:

- **SQLCipher database key (Rust core):** stored via platform keyring backends from
  the [keyring-rs](https://github.com/open-source-cooperative/keyring-rs) ecosystem
  — `apple-native-keyring-store` (iOS/macOS Keychain),
  `zbus-secret-service-keyring-store` (Linux Secret Service),
  `windows-native-keyring-store` (Windows Credential Manager),
  `android-native-keyring-store` (Android Keystore).
- **Nostr identity secret bytes (Flutter):**
  [`flutter_secure_storage`](https://github.com/juliansteenbakker/flutter_secure_storage)
  (iOS Keychain / Android Keystore / etc.).
- **In-memory secrets** are wiped with
  [`zeroize`](https://github.com/RustCrypto/utils/tree/master/zeroize).
